Project Deep Dive Presentation

What It Is

Map-based hiring platform with two flows:

  • Applicant mode for job discovery and application
  • HR mode for applicant review and decisions

Current Runtime

  • Frontend: http://localhost:3000
  • Backend API: http://localhost:5000/api
  • MongoDB: mongodb://localhost:27017/

Prepared on April 10, 2026

Node.jsExpressMongoDBMongoosedotenvCORS

System Architecture

Frontend Layer

  • App.js controls role and navigation flow
  • MapView.js shows city-specific startups on map
  • HRDashboard.js manages applicants and decisions
  • ApplyModal.js handles candidate submission

Backend Layer

  • server.js wires middleware and routes
  • routes/jobs.js has endpoint logic
  • models/Job.js defines Job + Applicant schema
  • Mongo stores jobs with embedded applicant records

Data Path

Browser UI -> Axios -> Express route -> Mongoose query/update -> MongoDB -> JSON response -> React state update.

Core API Endpoints

  • GET /api/jobs fetch all jobs
  • GET /api/jobs/:id fetch one job
  • GET /api/jobs/:id/applicants fetch applicants sorted by match score
  • POST /api/apply submit an application
  • POST /api/accept accept a candidate
  • POST /api/reject reject a candidate

Validation Rules

  • Required fields must exist
  • Duplicate application by same email blocked
  • Openings limit enforced before apply/accept

Response Pattern

  • Success: JSON payload + status 200/201
  • Errors: 400/404/500 with descriptive message

Match Score Logic

How It Works

  • Job tags and applicant skills are normalized (lowercase + trim)
  • Each required tag checks for overlap against submitted skills
  • Final score is rounded percentage of matched requirements

Formula

matchScore = round((matches / requiredTags) * 100)

Example: 2 matches out of 4 required skills => 50%

Database Model

Job Fields

  • title, company, location, salary
  • tags, description, openings
  • coordinates { lat, lng }
  • postedAt
  • applicants[]

Applicant Fields

  • name, email
  • skills[], experience
  • portfolio
  • matchScore
  • status: pending | accepted | rejected
  • appliedAt

Frontend Flows

Applicant Journey

  • Login -> Landing Page
  • Select city -> MapView
  • Select company -> Select role
  • Open JobDetails -> ApplyModal
  • Submit form -> API call -> Success message

HR Journey

  • Login as hr@test.com
  • Open HRDashboard
  • Select job -> fetch applicants
  • Review skills + match score
  • Accept or reject based on fit

Seed Data and Demo Value

Risks and Improvements

Current Gaps

  • Login is frontend-only simulation, no real auth
  • API docs do not fully include accept/reject routes
  • Duplicate/legacy files exist in frontend folder tree
  • Backend dev script with nodemon config needs cleanup

Next Upgrades

  • Add JWT auth and role-protected endpoints
  • Use request validation schema (Joi/Zod)
  • Improve match scoring with weighted skill relevance
  • Add test coverage for routes and integration flow

Summary

  • Strong prototype architecture with clear role-based UX
  • Map-first discovery is a differentiating product feature
  • Backend logic already supports practical HR workflows
  • With auth, validation, and cleanup, this can become production-grade
